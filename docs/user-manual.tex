\section{Introduction}

This manual describes how to install, run, and use Urban Scout. It is intended for someone other than the developers, such as a teaching assistant, evaluator, or new user, to be able to smoothly set up and interact with the application. Every feature available in the web interface is documented here.

Urban Scout is a unified mobile tourist assistant for Seoul. It combines place browsing, transit schedules, regional news and safety alerts, incident reporting, and itinerary planning into a single relational web application. The system supports two roles: regular users (tourists) and administrators (content managers).

\section{Installation and Setup}

To run the project locally, the following components must be available:
\begin{itemize}
    \item MySQL server (8.0 or later)
    \item Node.js (v18 or later) and npm
    \item a modern web browser
\end{itemize}

\boldheading{Step 1: Database setup.} Run the schema, seed, and stored procedure scripts against MySQL. The schema script creates the \texttt{UrbanScout} database and all of its tables, the seed script loads sample regions, places, users, admins, transit data, news, and itineraries, and the stored procedures script registers the 9 procedures used by the backend.

\begin{lstlisting}[language=Bash]
mysql -u root -p < database/schema.sql
mysql -u root -p UrbanScout < database/seed.sql
mysql -u root -p UrbanScout < database/stored-procedures.sql
\end{lstlisting}

\boldheading{Step 2: Backend setup.} Move into the server folder, copy the example environment file, fill in the MySQL credentials and a session secret, install dependencies, and start the Express server. The server listens on port 3001 by default.

\begin{lstlisting}[language=Bash]
cd server
cp .env.example .env
# edit .env with MySQL credentials and a session secret
npm install
npm start
# server runs on port 3001
\end{lstlisting}

\boldheading{Step 3: Frontend setup.} In a second terminal, move into the client folder, install the front-end dependencies, and start the Vite development server. Vite proxies API calls to the backend automatically.

\begin{lstlisting}[language=Bash]
cd client
npm install
npm run dev
# Vite dev server runs on port 5173
\end{lstlisting}

The backend and frontend must both be running for the application to work correctly. Once both servers are up, the app is reachable at \texttt{http://localhost:5173}.

\section{Logging In}

The login page supports access for both regular users and administrators.

\boldheading{Regular User Login.} A regular user selects the user mode on the login page and enters one of the seeded user email addresses.

\boldheading{Administrator Login.} An administrator selects the admin mode on the login page and enters one of the seeded admin email addresses.

The seeded accounts loaded by \texttt{seed.sql} are listed in Table \ref{tab:accounts}. Any of these emails can be used directly without a password since the system uses a simplified email-based login flow.

\section{Using the Application as a Regular User}

After logging in, regular users access the main pages through the bottom navigation bar. Each page below is reachable in one tap from the navbar except Place Detail, which opens after tapping a place card, and Itinerary Detail, which opens after tapping a trip card.

\subsection{Home Page}

After login, the home page shows a Seoul hero banner and three region cards. The user can subscribe or unsubscribe to each region, which controls what places, news, and alerts they see across the rest of the app.

\begin{table}[H]
    \centering
    \begin{tabularx}{\textwidth}{lX}
        \toprule
        \textbf{Element} & \textbf{What it does} \\
        \midrule
        \texttt{Subscribe} button on a region card & Adds that region to your feed. \\
        \texttt{Subscribed} button (filled blue) & Already following this region. Tap again to unsubscribe. \\
        \texttt{Logout} button (top right) & Ends the session and returns to the login page. \\
        \bottomrule
    \end{tabularx}
\end{table}

\subsection{Places}

The Places page lists places across the user's subscribed regions. Filter pills choose between Landmarks, Eateries, Transit, or All. The region dropdown narrows the list to a single region.

\begin{table}[H]
    \centering
    \begin{tabularx}{\textwidth}{lX}

        \textbf{Element} & \textbf{What it does} \\
        \midrule

        Region dropdown & Narrows the list to a single region. \\
        Place card & Tap to open the place detail page. \\
        \bottomrule
    \end{tabularx}
\end{table}

\subsection{Place Detail}

The Place Detail page shows full details of a place, including a map and the nearest transit stations. Users can add the place to one of their itineraries from this page.

\begin{table}[H]

        Nearby Transit Stations & Lists transit stations within walking distance and the distance in meters. \\
        \texttt{Add to Itinerary} dropdown plus \texttt{Add} button & Pick one of your trips and add this place as a stop. \\
        \bottomrule
    \end{tabularx}
\end{table}

\subsection{Transit}

The Transit page shows the three transit routes and a station picker. Selecting a station loads its arrivals, departures, the routes that serve it, and any nearby landmarks or eateries.

        Serving Routes badges & Show which transit lines stop at the selected station. \\

        Departures table & Same for departures. \\
        Nearby Places cards & Tappable cards for landmarks and eateries within walking distance. \\
        \bottomrule
    \end{tabularx}
\end{table}

The News page shows news posts for the regions the user is subscribed to. Posts can carry an attached safety alert.

\section{Using the Application as an Administrator}

Admins log in through the admin toggle on the login page. Each admin is assigned to one region and can only manage content for that region. The bottom navbar gives admins three direct tabs (Dashboard, Reports, News), and the dashboard stat cards link to the other admin pages.

\subsection{Admin Dashboard}

The Admin Dashboard gives an overview of the assigned region. Stat cards link to other admin pages, and the recent pending reports preview shows the three most recent unreviewed reports for a quick look.

\begin{table}[H]

\section{All Supported Functionalities}

The pages described above cover the main functionalities supported by Urban Scout:
\begin{itemize}
    \item user login
    \item admin login
    \item browsing regions and subscribing or unsubscribing
    \item browsing places by region or by type (landmarks, eateries, transit)
    \item viewing place details with map and nearby transit stations
    \item viewing transit routes and station schedules with arrivals and departures
    \item viewing nearby places from a transit station
    \item reading regional news and safety alerts
    \item submitting incident reports
    \item viewing user itineraries and managing items
    \item creating and deleting itineraries
    \item verifying or rejecting incident reports as an admin
    \item creating news posts with optional safety alerts as an admin
    \item adding new places and their subtypes as an admin
    \item adding transit routes and linking station-vehicle stops as an admin
\end{itemize}

Each of these functions interacts with the relational database through dedicated backend endpoints, and the interface is structured so that all of them are reachable within a few clicks.

\section{Tips for Smooth Usage}

For the smoothest experience:
\begin{itemize}
    \item Ensure the database is fully loaded with the seeded data.
    \item Ensure both backend and frontend servers are running.
    \item Use the seeded email accounts provided with the project.
    \item Keep the backend connected to the correct MySQL database.
    \item Refresh the page after backend or database changes if needed.
\end{itemize}
